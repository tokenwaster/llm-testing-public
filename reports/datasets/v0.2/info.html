What this is

A benchmark harness that runs the same 26 tasks against every model — local (LM Studio) and cloud (Claude subscription CLI, OpenRouter) — and records timing, tokens, cost and retries for each one. Currently suite v0.6.16, with 7 models across 6 runs in the live dataset.

Every task is a brand-new conversation: exactly one user message goes to the model (plus tool round-trips inside a tier-2 task), never anything carried over from a previous task. Transcripts record the message count and roles per request, so isolation is auditable rather than merely asserted. Runs are sequential by default, because local models share one GPU and parallel requests would corrupt the timing.

Failing is fine. The hard tasks exist so models have headroom to grow into. A model that solves 128k-token recall today tells you more than ten models tied at 1.00 on easy tasks.

Scoring lanes

Three ways a task can be graded. The lane is a property of the task, not the model.

LaneUsed byHow the score is produced
pytestcoding-python · agentic A checker test-suite runs against the model's workspace. Score = fraction of tests that pass.
answerreasoning · long-context · math The final ANSWER: line is matched exactly, numerically (with tolerance), or by regex. Score is 1 or 0.
responseinstruction-following · hallucination · extraction · tool-use The model's whole reply is saved and a checker inspects it directly — format/constraint adherence, JSON field accuracy, grounded-answer vs correct abstention, or the right prompt-based tool call. Score = fraction of checks that pass.
webappone-shot-apps The model's app.html is driven by headless Chromium (Playwright), which asserts real behavior. Score = fraction passing. A render task may cap this lane and hand the rest to a human — see Human-graded craft.

Tiers

Tier 1 — single-shot. Prompt in, response out. Every model can attempt these.

Tier 2 — agentic. The model gets list_files / read_file / write_file / run_python in a private workspace and iterates until it's done. Models with supports_tools: false skip these automatically. Claude models run through Claude Code's own agent harness instead of the generic tool loop — transcripts record which harness was used, so the comparison stays honest.

What the numbers mean

MetricMeaning
Score 0–1. Pass fraction for pytest/webapp lanes; 1 or 0 for the answer lane. The leaderboard averages a model's latest score per task. One render task splits its score with a human reviewer — see Human-graded craft.
Wall Total elapsed time for the task including every retry. The clock never lies — a model that needed three attempts pays for three attempts.
TTFT Time to first token (streaming only). How long the model thinks before it starts speaking.
tok/s Generation speed: output tokens ÷ generation time.
prefill tok/s Prompt-processing speed: input tokens ÷ time-to-first-token. This is what long-context tasks stress.
Tokens in/out Taken from the provider's own usage field — never estimated.
Cost Billed cost when the gateway reports it (OpenRouter), otherwise computed from the list pricing in the model's yaml. Which basis was used is recorded per result.
Retries Extra attempts consumed. Errors AND format failures (no ANSWER line, no code block) both burn an attempt.
Tries/pass Attempts spent per perfect (1.0) result — an efficiency measure. Lower is better; a model that one-shots everything sits at 1.0.
Attributed score The score with failures that were not the model's fault (harness bugs, infrastructure errors) removed. The gap between raw and attributed score is how much the harness cost that model.
Cold start Local models only: the measured lms load time before timing begins.

Run statuses

What a result's status means, before any scoring is applied.

StatusMeaning
ok The model responded and the result was scored normally.
error Every attempt failed (timeout, connection, API error, or no usable output). Scored 0 — a real failure, and it counts.
max_turns A tier-2 model used up its agent turns before finishing. Its workspace is still graded as-is, so a partly-finished job can still earn partial credit.
⏸ usage limit A Claude subscription cap (5-hour / daily / weekly) was hit mid-run. The in-flight task is dropped unscored — deliberately NOT a zero — that model's remaining tasks are skipped, and the reset time is recorded. Re-run after the reset and the gap fills in.
skipped The task is tier 2 and the model has supports_tools: false. No result is recorded.

Failure taxonomy

Every non-passing result is classified, and each class is attributed — to the model, to the harness, or to infrastructure. This is what separates "the model got it wrong" from "we broke it", and it's what the attributed score corrects for.

ClassBlameMeaning
pass clean scored at or above the pass threshold
partial model partial credit — some checks passed, some failed
wrong-answer model produced an answer; it was incorrect
fell-for-trap model gave the memorized classic answer — the task's twist changes it (pattern-matched instead of reading)
retrieval-miss model found the wrong value in the long context (retrieval / recency / aggregation error)
checker-fail model code ran but failed the checker's tests
infinite-loop model the model's code ran too long and the checker timed out — almost always an infinite loop
forbidden-construct model used a construct the task forbids (e.g. eval / the re or json module)
rumination-spiral model thought until the token budget was gone, never produced a usable answer
incomplete-output model answer was cut off when the token budget ran out
format-miss model produced output but not in the required format (no ANSWER/code/html block)
agentic-max-turns model ran out of agent turns before finishing
suspect-answer-key harness multiple models gave this same answer — the answer KEY is the likely fault, not the model
budget-artifact harness hit a token cap BELOW the fleet budget — an unfair-budget config bug, re-run before it counts
checker-error harness the checker itself errored (no tests ran) — a harness/environment problem, not the model
transport-drop infra the gateway/connection dropped — a flaky pipe, not model capability
rate-limit infra provider rate-limited the request
timeout infra the request timed out
context-overflow known-limit prompt exceeded the model's usable context window — expected, config-visible
unknown-error infra errored for an unclassified reason

Task fit

On the overview, every model is classified per category so you can pick the right model for a job rather than just crowning one winner:

  • Top quality — the highest score in that category (ties share the crown).
  • Value pick — the fastest generator that still clears the capable threshold.
  • Also capable — clears the threshold, but isn't the best or the fastest.
  • Below par — doesn't clear the threshold for this category.

The thresholds live in directives.yaml and are presentation, not scoring: edit them and the classification updates on the next report regeneration, with no suite version bump.

Reading the charts

Efficiency frontier

Average score against average output tokens per task. Up is better, left is cheaper. A model that answers correctly in 200 tokens beats one that ruminates for 2,000 to reach the same place — the top-left corner is where you want to live.

Rank across suite versions (bump chart)

How each model's rank moved as the test suite evolved. Hover a node to highlight that model across every version; models tied at a rank share the node. Rank shifts here usually mean the tests got harder, not that a model got worse.

Value scatters (score vs cost / speed)

The leaderboard ranks on score alone, which saturates at the top — the best few sit within a couple of percent. These scatters separate them on what actually differs: cost (API dollars to run the whole suite; a local model's dollar is just electricity, so that chart is API-only) and speed (tok/s, with a local/remote toggle). The dashed line is the Pareto frontier; a dimmed dot is dominated — some other model scores at least as high while costing less or running faster, so it is never the rational pick. Hover shows every model under the cursor.

Colours and dots

Every chart on the overview shares one colour per model, so a model is the same colour everywhere. Charts are dots rather than lines because runs are discrete measurements, not a continuous signal — connecting them would imply a trend that isn't there.

Methodology guarantees

  • Content-hashed tasks. Every result records a hash of the task definition. Edit a task and the longitudinal report flags the break instead of silently mixing versions.
  • Every run counts — repeats aggregate. Test a model on a task more than once in a version and the score becomes the mean of those runs, not the newest: a second opinion fleshes the number out instead of overwriting it, and the σ beside it is the spread. One model·task measured once is simply the mean of one. Runs that never produced a score (crash, spiral, DNF) stay out of the mean. A rescore still supersedes — it re-grades the same run in place rather than adding one — and a genuinely botched run is deleted on /manage rather than averaged in.
  • Equal budget. Every non-Claude model gets the same max_tokens. A model that died at a smaller budget is re-run before its zeros are allowed to count.
  • Local models run fully on the GPU (fixed 2026-07-18). Local models are loaded with --gpu max and a context window sized to each group of tasks. Before this, LM Studio's default "auto" offload left layers on the CPU with VRAM free (gemma-4-31b: 8 tok/s at 17% GPU), and one 128k-context task forced the whole run into a window too big to fit — so every task in the run crawled. Now the short-context tasks load in a window that fits and run on the GPU (~57 tok/s, 82%), and only genuine long-context tasks that exceed the card pay the cost. This changes speed, not scores — same prompt, budget, and weights. So tok/s, wall-clock, score/min, and energy for large local models step up on this date and are not comparable to earlier runs; score-based views (leaderboard, version-over-version) stay valid. Each run records its load plan in model_meta.json.
  • Reference-verified tasks. A new or changed task ships only once a known-good implementation scores 1.0 and an empty or trap submission scores 0. This is what catches checkers that hand out free credit for doing nothing.
  • Honest wall time. Retries are counted in it. Tokens come from the provider's usage field, never estimated.
  • Append-only ground truth. runs/ holds full transcripts, metrics, scores and the model's actual workspace. Reports are a pure function of it — delete reports/ and regenerate at any time.

Contamination & memorization

The hard question for any benchmark is whether a model scores well because it reasons or because it has seen the answer in training. We can't prove a private model's training set, so instead of claiming immunity we design the tasks to make memorization not pay, and we tell you exactly how:

  • Original constructions, not public sets. The tasks are written for this suite. They are not lifted from MMLU, HumanEval, GSM8K, or any public leaderboard, so a model that memorized those gains nothing here.
  • Twisted classics. Several reasoning tasks take a famous puzzle and change the constraint that matters (the reversed river-crossing, the Monty Fall variant). A model regurgitating the well-known answer scores zero — the memorized response is now the wrong one. That is a contamination detector, not just a question.
  • Behaviour over recall. The one-shot-app and agentic tasks are graded on what the model builds — a Playwright suite drives the generated app, a pytest checker runs the model's code. You cannot memorize your way through "make this maze solvable"; the artifact either works or it doesn't.
  • Fresh long-context payloads. Long-context tasks assemble their haystack at generation time, so the specific facts to retrieve aren't a fixed string sitting in any crawl.
  • Content-hashed and dated. Every result records the task's content hash and the run date, so if a task ever leaked and were quietly revised, the break is visible rather than silent — and you can see whether a model was tested before or after a given model's training cutoff.

What we do not claim: that any closed model definitely never saw a task. We claim the tasks are built so that seeing them helps little, and that the twisted-classic scores are direct evidence a model is reasoning rather than reciting.

Sample size & how much to trust a number

Be a skeptic — here is exactly how much data is behind each figure, stated plainly rather than buried.

  • This dataset holds 6 runs across 7 models and 26 tasks, for 182 scored model·task cells totalling 340 individual graded trials.
  • Trials per cell. 69% of cells are backed by more than one run; the rest are a single trial. A cell run more than once shows its spread (σ) and its mean is over every scored run — see the aggregation rule above.
  • The headline band. Each model's score carries a 95% confidence band (±1.96·SE) computed across the task set — treating the 26 tasks as a sample. Two models whose bands overlap are not distinguishable on this suite; the overview marks them tied (≈). This is why the top cohort reads as a near-tie rather than a clean ranking.
  • Small by design, honest about it. 26 tasks is a deliberately small, hand-verified set, not a scraped thousand. Only a handful currently separate the frontier models (see task discrimination) — the number the suite is least sure of is who is #1 among the top few, and it says so rather than manufacturing a decisive gap.

Caveats — read this before trusting a number

Every benchmark has edges where the number means less than it looks like it does. These are ours.

Tier-2 isn't strictly apples-to-apples.

Claude models run agentic tasks through Claude Code's own agent harness (its native tools, in the task workspace); every other model runs the harness's generic tool loop. Both are recorded — transcripts carry agent_harness — but a tier-2 gap between Claude and a local model is partly a gap between two agent scaffolds, not purely between two models.

A cloud model's serving host can drift between runs.

OpenRouter routes to whichever provider it likes, and providers differ in precision (fp8 / fp4 / unknown). Each result records who actually served it and at what quantization — shown as via <host> (quant). If a model's score moves between runs, check that column before blaming the model.

Cost is an estimate, not an invoice.

Most cost figures are computed from a list price captured when the model was registered — a snapshot, not a live rate. Only results where the gateway reported an actual billed amount (marked ✓) are authoritative. Published rates change, and a gateway routes the same model to different upstream hosts at different prices. See Pricing below for exactly how many of these numbers are snapshots.

The answer lane is all-or-nothing.

A right answer in the wrong format scores 0. That's deliberate — following the output contract is part of the task — but a 0 here doesn't always mean the model didn't know. Any 0 whose expected value is sitting inside the ANSWER line gets flagged [FORMAT-MISS] so it can be reviewed rather than quietly averaged in.

Old archived datasets are not comparable to the current one.

Several agentic checkers used to hand out free credit for doing nothing (a no-op scored up to 0.80 on ag-007). Those floors were removed in v0.5.5 and v0.5.6, and live results were rescored — but archived datasets keep their original scores, by design. Compare within a dataset, not across them.

Timing-scored tasks are calibrated, not absolute.

ag-006 grades an algorithmic speedup, and the naive solution is orders of magnitude slower than the optimised one. But an absolute wall-clock budget would measure how busy the machine was rather than how good the model was — we learned that the hard way when a correct 0.3s submission scored zero on a loaded box. The budget now times a fixed reference workload in the same subprocess and scales itself to the machine's current speed, so the verdict holds under load. The harness also refuses to rescore while a run is executing, because that contention corrupts the very budgets it is measuring.

Speed numbers only mean something on identical hardware.

Every run stores its own hardware fingerprint. If two runs were measured on different rigs, their tok/s are not comparable — the fingerprints are shown below so you can check.

Pricing — read this before quoting a cost

Every dollar figure on this site is an estimate of what a run would have cost, not a bill. Four things are true at once, and all of them matter:

  • The list price is a snapshot, not a live rate. A model's $/Mtok is captured in its yaml when the model is registered and does not update itself. Providers change their prices; this site does not notice. 0 of 1 cost figures (0%) are computed from that snapshot.
  • Only billed figures are authoritative. When the gateway reports what it actually charged, we use that and mark it ✓. That is 0 of 1 results (0%) — the rest are arithmetic.
  • The same model is served by different providers at different prices. OpenRouter routes to whichever upstream it likes, and they differ in price and in quantization. Across this dataset the catalog was served by 0 different hosts. A model's cost — and its score — can move between runs without the model changing at all.
  • Claude subscription runs report the API-equivalent price, not what the subscription actually costs you. Treat those as "what this would have cost on the API", not as spend. The estimate applies Anthropic cache pricing — a re-read (cache-hit) input token is billed at 0.1× and a cache write at 1.25× the base rate — so a multi-turn agentic run isn't charged full price for re-sending its context each turn. Runs recorded before this was added lack the cache breakdown and read high until the model is re-run.

"Free" models are on a promotion, not a price

OpenRouter publishes promotional variants with a :free suffix on the model id (tencent/hy3:free). Their $0 is a true record of what those runs were billed — but it is a promotion with an end date, not a rate you can plan around. Treating it as a price is how a benchmark starts recommending something that costs money tomorrow.

So the site separates two very different zeroes:

  • A local model's $0 is durable — it is measured GPU electricity (see below). It cannot expire.
  • A :free gateway model's $0 is temporary — shown with a ⏳ free-tier marker, and deliberately excluded from the "cheapest that works" recommendation on Task fit, where it appears separately as "free now". Speed does not expire, so it can still win "fastest that works".

This is not hypothetical: a price refresh of this catalog found four free tiers had already ended and one model's price had nearly doubled since it was registered. Prices carry a pricing_asof date for exactly this reason, and re-reading the live catalog is a one-command operation — but until it is re-read, every list price here is a snapshot of the day the model was registered.

Local models are not free — they just bill you differently

No money goes to a provider, so a local model's $ column is ~0. The wall socket still charges you. The harness samples the GPU throughout every local run, so this is measured, not modelled: peak/average power, and total watt-hours. The Power cost column turns that into money at your rate — currently $0.13 per kWh, set in directives.yaml (change it and the next report regeneration picks it up; it's presentation, so no version bump).

Two limits you must keep in mind, or this number will flatter local models:

  • It is GPU-only. It excludes the CPU, RAM, board, PSU losses and cooling — so it is a lower bound on what the machine actually pulls from the wall, not a utility bill.
  • It is marginal, and ignores the hardware. It says nothing about the cost of the GPU itself. Amortised over any realistic life, the card dwarfs the electricity it will ever burn running this suite. A local model is cheap per run; it was not cheap to own.

Because of that measurement, a local model's Cost / run is its electricity, marked ⚡ — not $0. It used to read "free", which is exactly the unexamined assumption this project exists to puncture. Score / $ follows: a local model is now ranked on value instead of hiding behind "free".

But the comparison is not symmetric, and you must hold this in your head: a cloud model's price includes the provider's hardware, power, staff and margin. A local model's ⚡ figure includes none of yours — not the GPU, not the rest of the box. So "local scores thousands of points per dollar and Opus scores less than one" is true as marginal cost and misleading as total cost of ownership. The local number is a floor; the cloud number is a price. Compare locals to locals with confidence, and locals to cloud with your eyes open.

Bottom line: cost here is sound for order-of-magnitude comparison between models in the same dataset. It is not sound for budgeting, for quoting a vendor's current price, or for comparing costs across dates. If you need a real number, go to the provider.

Where the raw data lives

runs/ is append-only ground truth. Reports are a pure function of it, so you can delete reports/ and regenerate at any time — and you can audit any number on this site down to the raw exchange that produced it.

PathWhat's in it
runs/<run>/run.jsonRun manifest: suite version, models, task hashes, hardware fingerprint, and any pause reason.
…/<model>/<task>/transcript.jsonl Every request and response, verbatim — including each retry and every tool round-trip.
…/metrics.jsonTiming, tokens, cost, retries, turns, and the per-attempt breakdown.
…/score.jsonThe score, who produced it, and the checker's own output.
…/workspace/The files the model actually wrote. For agentic tasks this is what gets graded; for one-shot apps it holds the app.html that was driven by the browser.

You never need the filesystem to look: every completed task in the run log carries a files → link, and the /data browser walks the whole tree in the page. Generated app.html files open live, so you can play with what the model built.

Versioning & datasets

SUITE_VERSION versions the test dataset, not the code. A minor bump means the tests or the methodology changed, and the old data is archived first so live reports always show exactly one coherent dataset. A patch bump is a scoring fix or a presentation change within a dataset — patches never archive, because the archive key is major.minor, so every 0.6.x run aggregates together. Use the dataset selector in the top-right of the overview to view an archived set, rendered with its own task-definition snapshot.

Changelog

All notable changes to the test suite and the harness.

Versioning policy. SUITE_VERSION versions the *test dataset*, not the code:

  • Minor/major bump (0.4 → 0.5) — the tests or the methodology changed. The old data is archived first (harness archive --as 0.4) so live reports always show exactly one coherent dataset.
  • Patch bump (0.5.6 → 0.5.7) — scoring fixes and presentation changes *within* a dataset. Patches never archive: the archive key is major.minor, so every 0.5.x run aggregates into the same "0.5" dataset.
  • No bump — run-engine, server, and UI changes that don't affect what is tested or how it is scored.

Scores are aggregated as the mean of every scored run per model·task, so re-running a model fleshes its number out rather than replacing it; unscored runs (crash / spiral / DNF) stay out of the mean.
